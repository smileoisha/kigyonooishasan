# スキル：面談後フォローアップ (/follow-up)

## 使い方
```
/follow-up [顧客名]
```

---

## 処理フロー

### Step 1：データ収集（自動）

以下を並行して取得する。

#### 1-A. 顧客情報（get_customer）
```
get_customer で [顧客名] を検索し、以下を取得：
- customer_id
- 顧客名（正式名称）
- 契約プラン
- 直近面談履歴
```

#### 1-B. 面談文字起こし（Gmail）
```
Gmail を検索：
  from:gemini-notes@google.com
  件名または本文に [顧客名] を含む
  直近1件を取得
```
- 取得できない場合：「文字起こしが見つかりません。Gmailを確認してください。」と表示して続行

#### 1-C. 未解決の困りごと一覧（get_customer_concerns）
```
get_customer_concerns(customer_id="{1-Aで取得したID}", status="open")
```
- 0件の場合：Step 2の⑤はスキップ、出力Bはスキップ

#### 1-D. 次回面談日（Google Calendar）
```
Google Calendar を検索：
  [顧客名] を含むイベント
  本日以降 直近1件
```
- 取得できない場合：「（次回日程未定）」を使う

---

### Step 2：Claude 要約生成

文字起こし全文 + 未解決困りごと一覧を入力として、以下5観点で要約を生成する。

#### ① 決定事項
- 面談中に合意・確定した事項を箇条書きで抽出
- 「〜することにした」「〜で行く」「〜に決めた」等の表現を検知
- 曖昧なものは含めない（未決事項に回す）

#### ② アクションアイテム（院長側 / 顧客側）
- 誰が・何を・いつまでに、を明確に分離
- 「やっておきます」等の曖昧な発言も拾い、担当を明示
- 期限が不明な場合は「次回面談まで」とする

#### ③ 顧客の感情・温度感の変化
- 面談開始時と終了時の温度感の差を読む
- 不安・安心・納得・困惑・前向き等を発言パターンから推測
- 例：「融資額に不安 → 数字で説明後に納得感」
- 読み取れない場合は「温度感の変化 読み取れず」と記載

#### ④ 未決事項
- 結論が出なかった論点
- 「次回までに確認」「要検討」「持ち越し」に該当するもの

#### ⑤ 困りごと解決マッチング
- 未解決困りごとの各投稿本文と、面談の文字起こし内容を突合
- 面談の中で扱われた困りごとを特定し、解決内容（どう整理・解決したか）を自然な文章で生成
  - 例：「資金繰りの不安について、直近3か月のキャッシュフロー予測を確認し、
         当面の支払いに支障がないことを数字で確認。当座の불安は解消。」
- マッチしなかった困りごとは**スキップ**（誤マッチ防止。面談で触れていなければ転記しない）
- マッチした場合：困りごとID・タイトル・生成した解決内容テキストを記録する

---

### Step 3：出力生成（3つ）

#### 出力A：フォローアップメール下書き（Gmail draft_create）

```
件名：本日の面談まとめ・次回についてのご連絡

※ このメールの宛先を [顧客名] さんのメールアドレスに変更してから送信してください ※

[顧客名] 様

本日はお時間をいただきありがとうございました。
確認した内容と次のステップをまとめてお送りします。

【本日の決定事項】
（① から生成）

【次回までの宿題】
・院長：（② の院長側）
・[顧客名]様：（② の顧客側）

【次回面談】
・日時：[1-Dで取得した日時]

---
本日の面談で触れた「困りごと」について、解決の記録を追記しました。
お手すきの際にご確認いただき、解決済みと感じるものは
「解決ボタン」を押していただけると助かります。

↓ 困りごと投稿ページ
https://task-manager-a5x.pages.dev/concerns.html

引き続きよろしくお願いいたします。
企業のお医者さん　菊地 勇紀
```

- **下書き保存のみ**。送信しない。

#### 出力B：困りごとDB への解決内容転記（⑤でマッチした件数分）

⑤でマッチした困りごとそれぞれに対して以下を実行：

```
PATCH https://task-manager-a5x.pages.dev/api/admin/concerns
Content-Type: application/json

{ "id": "[困りごとID]", "resolution": "[⑤で生成した解決内容テキスト]" }
```

**重要：**
- `status` は送らない（open のまま維持する）
- 顧客が内容を確認→自分で解決ボタンを押す → その時に resolved になる
- これが「教育・自立支援」の設計であり、院長が resolved に変えてはならない

#### 出力C：顧客ナレッジへの会議記録保存（create_customer_meeting）

```
create_customer_meeting({
  customer_id:  "[1-Aで取得したID]",
  date:         "[面談日付 YYYY-MM-DD]",
  title:        "[面談タイトル]（例：公庫面談練習・月次レビュー等）",
  summary:      "（①②③④をまとめた要約。決定事項・アクション・温度感・未決事項を含む。3分で振り返れる粒度）",
  action_plan:  "【院長】\n- ...\n\n【[顧客名]様】\n- ...",
  issues:       ["経営課題1", "経営課題2"],
  next_actions: ["次回アクション1", "次回アクション2"]
})
```

- `get_customer` の返却フィールド（`aiSummary` / `actionPlan` / `issues` / `nextActions`）と1対1で対応
- store の `customer.meetings[]` に追記され、knowledgeテーブルへ自動同期される
- `create_knowledge` は使わない（customer_meeting の仕組みに乗せる）
```

---

### Step 4：院長への確認表示

3つの出力がすべて完了したら、以下の形式でまとめを表示する：

```
─── /follow-up 完了（[顧客名]・[面談日付]） ───

■ メール下書き → Gmail に保存済み
  件名：本日の面談まとめ・次回についてのご連絡
  ※ 内容を確認してから送信してください

■ 困りごと解決内容 → [N]件を転記済み
  （0件の場合：「面談で扱われた困りごとなし」）
  - [困りごとタイトル or 本文冒頭30文字]
  - ...

■ 会議記録 → ナレッジに保存済み
  タイトル：[YYYY-MM-DD] 面談記録

修正が必要な箇所はありますか？
```

## エラー・例外処理

| 状況 | 対応 |
|------|------|
| 文字起こしメールが見つからない | 警告を表示してStep 2以降は「文字起こしなし」として続行。メール下書きはスキップ |
| 未解決困りごとが0件 | ⑤をスキップ、出力Bをスキップ |
| 顧客名が一致する顧客が複数いる | 候補を列挙して院長に選択を求める |
| Google Calendarで次回日程が見つからない | 「（次回日程未定）」として続行 |
| Zoom文字起こしの場合（Meet以外） | 「Zoomクラウド録画からの自動取得は未対応。文字起こしテキストを直接貼り付けてください」と案内 |

---

## 注意事項

- **送信はしない**。メール下書きは必ず院長が確認してから送信
- **status は変えない**。resolution 書き込みのみ。顧客が自分で解決ボタンを押す
- **誤マッチは転記しない**。面談で明確に扱われた困りごとのみマッチングする
- **要約は簡潔に**。3分で振り返れる粒度。Geminiの長文要約スタイルは踏襲しない

## 参考：API仕様

### concerns解決内容転記
```
PATCH /api/admin/concerns
{ "id": "xxx", "resolution": "テキスト" }
→ status: open のまま resolution だけ書き込まれる
```

### 顧客情報取得
```
get_customer([顧客名]) → customer_id, name, plan, 面談履歴
get_customer_concerns(customer_id, status="open") → 未解決投稿一覧
```

### 会議記録作成
```
create_customer_meeting(customer_id, date, title, summary, action_plan, issues[], next_actions[])
→ POST /api/admin/meetings 経由で store に追記 + knowledge 同期
→ 戻り値: { meeting_id, knowledge_id, title, date }
```
