## 2. D1テーブル設計

### `customer_concerns` テーブル

```sql
CREATE TABLE customer_concerns (
  id TEXT PRIMARY KEY,              -- UUID v4
  customer_id TEXT NOT NULL,        -- 既存customersテーブルのID
  email TEXT NOT NULL,              -- Cloudflare Accessから取得したメール
  body TEXT NOT NULL,               -- 困りごと本文（自由記述）
  urgency TEXT NOT NULL DEFAULT 'normal',  -- 'normal' | 'urgent'
  status TEXT NOT NULL DEFAULT 'open',     -- 'open' | 'resolved'
  created_at TEXT NOT NULL,         -- ISO 8601
  updated_at TEXT NOT NULL,         -- ISO 8601
  resolved_at TEXT,                 -- 解決済みになった日時
  auto_resolved INTEGER NOT NULL DEFAULT 0  -- 1=自動解決
);

CREATE INDEX idx_concerns_customer ON customer_concerns(customer_id);
CREATE INDEX idx_concerns_status ON customer_concerns(status);
CREATE INDEX idx_concerns_email ON customer_concerns(email);
```

### 顧客判定キー

- Cloudflare Accessの `CF-Access-JWT-Assertion` ヘッダーからメールアドレスを取得
- **⚠️ 既存の顧客データは `store` テーブルのJSON blob内に格納されている（独立テーブルではない）**
- 照合手順：`GET /api/data` でJSON blobを取得 → 顧客一覧（`data.customers`）からメールアドレスで検索 → `customer_id` を確定
- `customer_concerns` テーブルはJSON blobとは別の**独立テーブル**として新設（Cron Trigger・MCP直接クエリ・お客さん側からの直接書き込みが必要なため）
- **メールアドレスがJSON blob内の顧客データに未登録の場合はエラー**（顧問契約者のみ利用可）
- **実装前に `data.js` の顧客データ構造（フィールド名・メールアドレスの格納場所）を必ず確認すること**

## 3. API設計

### 認証（⚠️ 二系統混在に注意）

既存のタスクマネージャーは `auth.js`（localStorageベース）で管理者認証している。
困りごとフォームは Cloudflare Access JWT でお客さん認証する。
**この2つが同じWorkers内で混在する初のケース。**

| 系統 | 対象エンドポイント | 方式 | 認証情報 |
|------|-------------------|------|---------|
| 管理者認証 | `/api/data`, `/api/upload` 等（既存全部） | localStorage (`auth.js`) | `tm2_currentUser` |
| 顧客認証 | `/api/concerns*` のみ | Cloudflare Access JWT | `CF-Access-JWT-Assertion` → メール取得 |
| 管理者認証 | `/api/concerns/summary`（MCP用） | localStorage (`auth.js`) | 管理者のみアクセス可 |

```
顧客エンドポイント：
Cloudflare Access JWT → メール取得 → store JSON blobの顧客一覧でメール検索 → customer_id 解決
認証失敗 → 401
顧客未登録 → 403（「顧問契約者のみご利用いただけます」）
```

### エンドポイント

#### `POST /api/concerns` — 投稿作成

```
Request:
{
  "body": "売上が減ってきて不安です...",
  "urgency": "normal"  // "normal" | "urgent"
}

Response: 201
{
  "id": "uuid",
  "created_at": "...",
  "duplicate_warning": null | {
    "similar_id": "uuid",
    "similar_body_preview": "売上減少について...",
    "created_at": "..."
  }
}
```

**重複チェックの実装方針：**
- 投稿作成前に、同一顧客のstatus='open'の投稿を全件取得
- Claude API（Sonnet）に「新規投稿」と「既存未解決投稿リスト」を渡し、類似度を判定
- 類似ありの場合：`duplicate_warning` に情報を返し、フロント側で「似た内容がありますが投稿しますか？」と確認
- **フロントが確認後に再度POSTする際は `?force=true` クエリパラメータで強制投稿**

#### `GET /api/concerns` — 自分の投稿一覧 + 検索

```
Query:
  ?status=open|resolved|all (default: all)
  ?q=キーワード (任意・本文をLIKE検索)
Response: 200
{
  "concerns": [
    {
      "id": "uuid",
      "body": "...",
      "urgency": "normal",
      "status": "open",
      "created_at": "...",
      "updated_at": "..."
    }
  ]
}
```

**検索の実装：**
- `q` パラメータがある場合、`WHERE body LIKE '%キーワード%'` を追加
- D1のLIKE検索で十分（投稿数が少ないうちは全文検索エンジン不要）
- 顧客増加時に全文検索が必要になったら別途検討

#### `PATCH /api/concerns/:id` — ステータス更新

```
Request:
{ "status": "resolved" }

Response: 200
お客さんは自分の投稿のみ更新可。
管理者（院長・Masami）は全投稿を更新可。
```

#### `GET /api/concerns/summary` — MCP用（管理者のみ）

```
Query: ?customer_id=xxx
Response: 200
{
  "customer_name": "...",
  "open_concerns": [...],
  "resolved_concerns": [...],
  "total_open": 3,
  "last_submitted": "2026-05-20T..."
}
```

## 6. 緊急投稿の通知

### 通知方法

「今すぐ」選択時のみ、以下のいずれかで院長に通知：

**案A：メール通知（GAS不要・シンプル）**
- Workers内から Cloudflare Email Workers または外部メールAPIで院長宛にメール送信
- 件名：`【緊急】顧客名 — 困りごと投稿`

**案B：Slack通知（Slack Incoming Webhookを使用）**
- Slack Incoming Webhook URLをWorkers環境変数に設定
- 緊急投稿時にWebhookでSlackに通知

→ **推奨：案B（Slack通知）**。院長のSlackに通知が飛ぶ方が見落としにくい。メールは埋もれるリスクあり。

## 7. 自動解決（2週間ルール）

### 実装方法

**Cloudflare Workers Cron Trigger を使用**

```
cron: "0 0 * * *"  # 毎日0時（JST 9時）に実行
```

処理：
```sql
UPDATE customer_concerns
SET status = 'resolved',
    resolved_at = datetime('now'),
    auto_resolved = 1,
    updated_at = datetime('now')
WHERE status = 'open'
  AND created_at < datetime('now', '-14 days');
```

## 8. 面談後メール（議事録 + 解決確認）

### この部分は別フェーズ

以下の理由で、フォーム実装とは切り離す：

1. Zoom文字起こし → Claude要約 → GASメール下書きのパイプラインは独立した仕組み
2. Googleカレンダー連携（GAS）は別の技術スタックで動く
3. フォームが動いてからでないとメール内に「フォームで解決済みにしてください」リンクを入れられない

**→ フォーム実装完了後に着手**

## 9. 重複チェックのClaude API呼び出し

### Workers内での実装

```javascript
// 重複チェック（投稿前に実行）
async function checkDuplicate(newBody, existingConcerns, env) {
  if (existingConcerns.length === 0) return null;

  const response = await fetch('https://api.anthropic.com/v1/messages', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'x-api-key': env.ANTHROPIC_API_KEY,
      'anthropic-version': '2023-06-01'
    },
    body: JSON.stringify({
      model: 'claude-sonnet-4-20250514',
      max_tokens: 500,
      system: `あなたは投稿の重複チェックを行います。
新しい投稿と既存の未解決投稿を比較し、
内容が実質的に同じ（同じ問題・同じ悩み）かどうかを判定してください。
JSONのみで応答してください。
{"is_duplicate": true/false, "similar_id": "該当ID or null", "reason": "判定理由"}`,
      messages: [{
        role: 'user',
        content: `新しい投稿：\n${newBody}\n\n既存の未解決投稿：\n${existingConcerns.map(c => `ID:${c.id}\n${c.body}`).join('\n---\n')}`
      }]
    })
  });

  const data = await response.json();
  // レスポンスパース処理
}
```

### 環境変数

```
ANTHROPIC_API_KEY  — Claude API キー（Workers Secrets に設定）
```

## 備考

- **既存データはJSON blob方式**：`store` テーブルに全データがJSON blobで格納されている。`customer_concerns` は初の独立テーブル。`data.js` の `INITIAL_DATA` と顧客データ構造を実装前に必ず確認すること
- **認証が二系統混在**：既存（localStorage）と新規（Cloudflare Access JWT）が同じWorkers内で共存する。エンドポイントごとに認証方式を明確に分離すること
- フォーム画面のルーティングは既存のSPAルーティングに追加する形が自然
- Cloudflare Accessの保護設定は、Masamiまたは院長がダッシュボードで手動設定
