## 2. D1テーブル設計

### `customer_concerns` テーブル

```sql
CREATE TABLE IF NOT EXISTS customer_concerns (
  id TEXT PRIMARY KEY,
  customer_id TEXT NOT NULL,
  email TEXT NOT NULL,
  body TEXT NOT NULL,
  category TEXT DEFAULT 'other',
  urgency TEXT NOT NULL DEFAULT 'normal',
  status TEXT NOT NULL DEFAULT 'open',
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  resolved_at TEXT,
  resolution TEXT,
  resolution_knowledge_id TEXT,
  auto_resolved INTEGER NOT NULL DEFAULT 0
);
CREATE INDEX IF NOT EXISTS idx_concerns_customer ON customer_concerns(customer_id);
CREATE INDEX IF NOT EXISTS idx_concerns_status ON customer_concerns(status);
CREATE INDEX IF NOT EXISTS idx_concerns_email ON customer_concerns(email);
```

### カラム説明

| カラム | 型 | 説明 |
|--------|-----|------|
| id | TEXT | UUID v4 |
| customer_id | TEXT | customersテーブルのID |
| email | TEXT | Cloudflare Accessから取得したメール |
| body | TEXT | 困りごと本文（自由記述） |
| category | TEXT | 分類（cash_flow / no_money / expenses / hiring / marketing / repeat / anxiety / other） |
| urgency | TEXT | 'normal' or 'urgent' |
| status | TEXT | 'open' or 'resolved' |
| created_at | TEXT | ISO 8601 |
| updated_at | TEXT | ISO 8601 |
| resolved_at | TEXT | 解決済みになった日時 |
| resolution | TEXT | 解決内容テキスト |
| resolution_knowledge_id | TEXT | 関連ナレッジID |
| auto_resolved | INTEGER | 1=自動解決（Cron or fire-and-forget） |

### 顧客判定キー

- Cloudflare Accessの `CF-Access-JWT-Assertion` ヘッダーからメールアドレスを取得
- `customers` テーブルを直接クエリ：`SELECT id, name FROM customers WHERE email = ?`
- メールアドレスが未登録の場合は403エラー（顧問契約者のみ利用可）

## 3. API設計

### 認証（二系統混在）

既存のタスクマネージャーは `auth.js`（localStorageベース）で管理者認証。
困りごとフォームは Cloudflare Access JWT でお客さん認証。

| 系統 | 対象エンドポイント | 方式 | 認証情報 |
|------|-------------------|------|---------|
| 顧客認証 | `/api/concerns`（POST/GET）, `/api/concerns/:id`（PATCH/DELETE） | Cloudflare Access JWT | `CF-Access-JWT-Assertion` → メール取得 |
| 管理者認証 | `/api/concerns/summary`, `/api/admin/concerns` | localStorage (`auth.js`) | 管理者のみアクセス可 |

```
顧客エンドポイント：
Cloudflare Access JWT → メール取得 → customersテーブルでメール検索 → customer_id 解決
認証失敗 → 401
顧客未登録 → 403（「顧問契約者のみご利用いただけます」）
```

### エンドポイント

#### `POST /api/concerns` — 投稿作成

```
Request:
{
  "body": "売上が減ってきて不安です...",
  "urgency": "normal"
}
Response: 201
{
  "id": "uuid",
  "created_at": "...",
  "duplicate_warning": null | {
    "similar_id": "uuid",
    "similar_body_preview": "売上減少について...",
    "created_at": "..."
  }
}
```

**重複チェック：**
- 投稿作成前に、同一顧客のstatus='open'の投稿を全件取得
- Claude API（**Haiku** `claude-haiku-4-5-20251001`、max_tokens: 200）で類似度を判定
- 類似ありの場合：`duplicate_warning` に情報を返し、フロント側で確認
- **フロントが確認後に再度POSTする際は `?force=true` で強制投稿**

#### `GET /api/concerns` — 自分の投稿一覧 + 検索

```
Query:
  ?status=open|resolved|all (default: all)
  ?q=キーワード (任意・本文をLIKE検索)
Response: 200
{
  "concerns": [...]
}
```

※ GET時にfire-and-forgetで当該顧客の14日超投稿を自動クローズ（Cron補完）

#### `PATCH /api/concerns/:id` — ステータス更新

```
Request:
{ "status": "resolved" }
Response: 200
お客さんは自分の投稿のみ更新可。
管理者（院長・Masami）は全投稿を更新可。
```

#### `DELETE /api/concerns/:id` — 投稿削除

```
Response: 200
お客さんは自分の投稿のみ削除可。
管理者は全投稿を削除可。
```

#### `GET /api/concerns/summary` — MCP用（管理者のみ）

```
Query: ?customer_id=xxx
Response: 200
{
  "customer_name": "...",
  "open_concerns": [...],
  "resolved_concerns": [...],
  "total_open": 3,
  "last_submitted": "2026-05-20T..."
}
```

要約処理はサーバー側では行わず、データのみ返却。チャット側のClaudeが要約する方式。

#### `GET /api/admin/concerns` — 管理者用全件取得

管理者認証で全顧客の投稿を取得。管理画面用。

## 7. 自動解決（2週間ルール）

### 二重方式

**方式A：Cron Trigger（メイン）**

```
cron: "0 0 * * *"  # 毎日0時UTC（JST 9時）
```

全顧客の14日超未解決投稿を一括クローズ：

```sql
UPDATE customer_concerns
SET status = 'resolved',
    resolved_at = datetime('now'),
    auto_resolved = 1,
    updated_at = datetime('now')
WHERE status = 'open'
  AND created_at < datetime('now', '-14 days');
```

**方式B：GET時fire-and-forget（補完）**

`GET /api/concerns` 呼び出し時に、当該顧客の14日超投稿をクローズ。
Cronが何らかの理由で失敗した場合の補完として残す。
